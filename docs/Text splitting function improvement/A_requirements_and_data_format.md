# 設計書 A：要件定義 + データフォーマット

NEXUS 透明分割機構（Transparent Segmentation）設計書の第 1 部。

## 0. はじめに

### 0.1 この設計が解決する問題

42 万字以上の長編小説を 1 つの `<textarea>` に流し込むと、ブラウザ（Blink）のネイティブレイアウトコストで打鍵応答が数秒〜20 秒単位まで悪化する。これは JS 最適化では解決不可能なブラウザエンジンの制約であり、2026 年 4 月の計測・パッチ 5 発・uncontrolled 化によって確定した。

この設計は「NEXUS は長編に耐える縦書き執筆環境である」というプロダクトの根幹を守るため、**ユーザーから見て単一ファイルとして振る舞い、内部的には複数ファイルに分割された状態を維持するデータ層と UI 層**を定義する。

### 0.2 この設計の原則

1. **ユーザーは分割を意識しなくてよい**。機能として認識するのは任意、デフォルト挙動は「長編が勝手に軽く動く」。
2. **原稿を消さない**は機能ではなく契約である。設計のあらゆる層がこの契約を裏切らないよう組まれる。
3. **既存ファイルとの互換性を保つ**。単一 `.txt` 形式で書かれた作品は、そのまま開けて、そのまま保存できる。逆戻しも可能。
4. **章があってもなくてもよい**。章境界が明示されている作品、曖昧な作品、章概念がない作品、すべて同じ機構で扱う。
5. **執筆を中断させない**。自動分割・統合はユーザーの打鍵中には起きない。常に idle 時の非同期処理で行う。

### 0.3 用語

- **作品（Work）**: ユーザーが 1 つの物語として編集する単位。従来は 1 つの `.txt` ファイル。
- **セグメント（Segment）**: 内部分割の単位。1 つの作品は 1〜N 個のセグメントからなる。1 セグメントが 1 ファイル。
- **マニフェスト（Manifest）**: 作品の構造（セグメント順序・境界位置）を記述する JSON ファイル。
- **連結ビュー（Concatenated View）**: 全セグメントを結合した仮想の全文。ユーザーが見る文章はこれ。
- **セグメント境界（Segment Boundary）**: セグメント間の継ぎ目。物理的にはファイルの端だが、論理的にはユーザーから透明。
- **チャンク閾値（Chunk Threshold）**: 1 セグメントが超えたら分割、下回ったら統合を検討する文字数（デフォルト上限 6 万、下限 1 万）。

## 1. 機能要件

### 1.1 ユーザーから見える振る舞い

#### F-1. 連続編集

ユーザーが作品を開くと、全セグメントが連結された状態で編集できる。セグメント境界を意識せずに打鍵・カット・ペースト・ドラッグ・Undo/Redo できる。

#### F-2. 全文検索・置換

作品内の検索は全セグメントを対象に行う。検索結果へのジャンプは、該当セグメントを必要に応じて読み込みつつ、ユーザーには単一の全文内でのジャンプとして見える。

#### F-3. プレビュー・リーダー

編集不可モード（プレビュー・リーダー）は常に全セグメント連結で表示される。ユーザーは作品全体を通読できる。

#### F-4. エクスポート

PDF・EPUB・プレーン TXT のいずれも、全セグメントを連結した完全な 1 作品として出力する。賞応募用の全文 TXT はワンクリックで生成できる。

#### F-5. 章またぎの編集違和感ゼロ

- カーソルをセグメント境界をまたいで移動しても引っかからない
- セグメント境界で Backspace を押すと前セグメントの末尾に戻る
- 選択範囲がセグメントをまたいでも正しく機能する
- Undo/Redo はセグメントをまたいで動く（Undo 履歴は作品単位）
- IME 変換中にセグメント境界を踏んでも変換バッファは壊れない

#### F-6. 自動分割

1 セグメントがチャンク閾値上限を超えた場合、システムは idle 時に自動分割する。ユーザーには原則通知されない（通知の有無は設定で切り替え）。分割後もカーソル位置・選択範囲・Undo 履歴は保たれる。

#### F-7. 自動統合（将来拡張）

大幅削除で小さくなったセグメントは、隣接セグメントと統合する。これは Phase 3 以降の拡張機能とし、初期リリースには含めない。

#### F-8. 逆戻し（新→旧フォーマット変換）

セグメント化された作品を単一 `.txt` ファイルに戻す機能を常時提供する。NEXUS をやめて他エディタに移る場合も、作品は奪われない。

### 1.2 UI モード

#### F-9. 連続エディタモード（デフォルト）

全セグメントが縦一列（縦書きなら横一列）に連結して表示される。ユーザーは 1 つの長い原稿用紙を見ている感覚。パフォーマンス保証のため、画面外セグメントの DOM は遅延マウントされる（仮想化）。

#### F-10. 章タブモード

各セグメントが個別のタブとして表示される。タブ切替で他セグメントに移動。タブ間コピーペーストは連続編集と同じ違和感ゼロで動く。

設定で切り替え可能。作品ごとに記憶。

### 1.3 変換・検出

#### F-11. 意味境界の自動検出

セグメント分割時、システムは以下の優先順位で境界候補を探す：

1. **明示的章境界**：行頭が `■`、`第[0-9０-９一二三四五六七八九十百]+章`、`# ` （1つ以上の `#` で始まる Markdown 見出し）、青空文庫タグ `［＃大見出し］`・`［＃中見出し］`
2. **意味的境界**：空行 3 連以上（章の間に空行を多く入れる慣習）、青空文庫タグ `［＃改ページ］`
3. **段落境界**：空行 1 つ以上（最低限の段落境界）
4. **文境界**：`。` 直後（どうしても意味境界がない場合の最終手段）
5. **強制分割**：上記いずれも 4 万字以内に見つからなければ文字数で機械的に切る

いずれの場合も、文字列の途中（文中、単語中、ルビタグ内、青空文庫タグ内）で切ることは**絶対にない**（契約 C-6 参照）。

### 1.4 ユーザー操作

#### F-12. 手動分割

連続エディタ上で右クリック → 「ここで章を分ける」を選ぶと、カーソル位置でセグメント分割できる。これは自動分割より優先されるユーザー意図として記録される。

#### F-13. 手動統合

章タブモードでタブを右クリック → 「前の章と統合」「次の章と統合」で隣接セグメントと統合できる。

#### F-14. セグメントのリネーム

各セグメントには**表示名**を付けられる（章タイトルなど）。これは 連続エディタモードの境界部分に小さく表示される（設定で非表示可）。

#### F-15. セグメントの並べ替え

章タブモードではドラッグで順序を変更できる。順序変更はマニフェストを書き換えるだけで、セグメント本体には触らない。

## 2. 非機能要件

### 2.1 パフォーマンス

- **P-1**: 単一セグメントは 6 万字以下を目標とし、打鍵応答は全モードで体感 100ms 以下
- **P-2**: 作品オープン時の初期レンダリング（最初の 1 セグメント表示まで）は 500ms 以内
- **P-3**: 全文検索は 100 万字の作品で 3 秒以内
- **P-4**: エクスポート（全文 TXT）は 100 万字で 1 秒以内
- **P-5**: 自動分割は ユーザーの打鍵停止後 10 秒以上の idle 時にのみ開始する

### 2.2 破壊耐性（最優先）

- **D-1**: いかなるバグ、クラッシュ、電源断、書き込み中断が発生しても、**直前の確定状態**（最後に完了した保存）にロールバックできる
- **D-2**: 任意の時点の作品状態に、**最低 30 日間**戻せる（スナップショット機構）
- **D-3**: **ユーザーが手動でマークしたスナップショット**は無期限に保持される
- **D-4**: 同じ作品を複数の NEXUS インスタンスで同時編集しても、**どちらかの編集が失われることはない**（後勝ち・警告・ロックのいずれかで対処）
- **D-5**: ファイルシステムの部分障害（exFAT の書き込み失敗、ネットワークドライブの切断など）でも、作品全体が壊れることはない（1 セグメントの損傷に留まる）

### 2.3 互換性

- **C-1**: 既存の単一 `.txt` 作品は NEXUS を更新後もそのまま開ける
- **C-2**: 既存作品に自動分割を適用するかは**ユーザーの明示承認**が必要。自動的には変換しない
- **C-3**: 変換後の作品でも、ワンクリックで単一 `.txt` に戻せる
- **C-4**: `[METADATA]` ブロックは引き続き機能する
- **C-5**: 青空文庫タグ（`［＃…］`）・ルビ（`《》`）・フォントタグ（`{font:…}`）は分割の影響を受けない
- **C-6**: ルビ、青空文庫タグ、フォントタグ、`**強調**` のような**ペア構造のマークアップ**は、必ずセグメント内で閉じる。セグメント境界がこれらの内部に入ることはない

### 2.4 運用

- **O-1**: 分割機構は**永続機能**として Antigravity の基本設計に組み込まれる（オプションではない）
- **O-2**: ただしセグメント数が 1 の作品（＝分割されていない作品）は、従来通りの挙動になる（機構の存在がユーザーに見えない）
- **O-3**: すべての分割・統合・スナップショット操作はログに記録される（`.nexus-project/journal.log`）

## 3. データフォーマット

### 3.1 プロジェクトルートの構造

NEXUS の「プロジェクト」はユーザーが選んだフォルダ（Electron の dialog.openDirectory で選択）。その直下に作品ファイルと `.nexus-project/` 管理領域が並ぶ。

```
MyNovels/                          ← プロジェクトルート（ユーザー選択）
├── .nexus-project/                ← 管理領域（後述）
│   ├── version.json
│   ├── journal.log
│   └── snapshots/
├── 虚空の三叉.txt                  ← 単一ファイル作品（従来形式、そのまま編集可能）
├── ノチユ.nexus/                   ← セグメント化された作品（新形式）
│   ├── manifest.json              ← 作品マニフェスト
│   ├── segments/
│   │   ├── 00000_章立て_01.txt
│   │   ├── 00001_章立て_02.txt
│   │   └── ...
│   └── snapshots/                 ← 作品固有スナップショット
└── メモ.txt                        ← 通常のメモファイル
```

### 3.2 単一ファイル作品（従来形式）

既存の `.txt` ファイルは従来通り。何も変わらない。ファイルサイズが閾値以下なら自動分割されない。閾値を超えても、ユーザーが「この作品を分割する」を明示選択するまで単一ファイルのまま。

### 3.3 セグメント化作品（新形式）

拡張子 `.nexus` のディレクトリ（macOS では Bundle として振る舞わせるため `.nexus` を Info.plist に登録する将来オプションあり。初期リリースでは通常のディレクトリ）。

#### 3.3.1 manifest.json

```json
{
  "formatVersion": 1,
  "workId": "01936b9c-a3c5-78e4-9c2a-...",
  "title": "星（ノチユ）に願いを",
  "created": "2026-04-19T10:00:00+09:00",
  "modified": "2026-04-19T15:23:11+09:00",
  "metadata": {
    "tags": ["社会派ミステリー"],
    "target": "松本清張賞"
  },
  "segments": [
    {
      "id": "s-00000",
      "file": "segments/00000_プロローグ.txt",
      "displayName": "プロローグ",
      "charCount": 3240,
      "checksum": "sha256:a3f2c1...",
      "modified": "2026-04-19T15:23:11+09:00"
    },
    {
      "id": "s-00001",
      "file": "segments/00001_第1章_出会い.txt",
      "displayName": "第1章　出会い",
      "charCount": 42180,
      "checksum": "sha256:b4e3d2...",
      "modified": "2026-04-19T14:10:02+09:00"
    }
  ],
  "chunkThresholds": {
    "splitAbove": 60000,
    "mergeBelow": 10000
  },
  "uiPreferences": {
    "defaultMode": "continuous",
    "showSegmentTitles": true
  }
}
```

**フィールド説明**：

- `formatVersion`: マイグレーション用。フォーマット変更時にインクリメント
- `workId`: 作品固有 UUID v7。複製・移動しても追跡できる
- `title`: 作品タイトル。表示用
- `metadata`: 従来の `[METADATA]` ブロックに相当。タグ・応募先など
- `segments`: セグメント配列。**この配列の順序が作品の順序**
  - `id`: セグメント ID。ファイル名とは独立（リネームしても ID は不変）
  - `file`: マニフェストからの相対パス
  - `displayName`: UI 表示用
  - `charCount`: キャッシュされた文字数。実ファイルから再計算可能
  - `checksum`: ファイル内容の SHA-256。整合性検証用
  - `modified`: 最終更新時刻
- `chunkThresholds`: この作品の分割閾値（グローバル設定を上書き）
- `uiPreferences`: UI 設定（作品ごとに保存）

#### 3.3.2 セグメントファイル

```
虚空の三叉_01_オデッサの残光.txt
```

拡張子は `.txt`。中身はプレーンテキスト（UTF-8、LF 改行）。**従来の本文とまったく同じ形式**。ルビ・青空文庫タグ・フォントタグ含めて何も変わらない。

これにより：
- セグメントファイル単体を他エディタで開いて編集可能（緊急時の逃げ道）
- 外部ツール（grep、wc など）での処理が普通に通る
- NEXUS が壊れても、セグメントを全部結合すれば作品を復元できる

セグメントファイルは **`[METADATA]` ブロックを持たない**。メタデータは作品全体に 1 箇所（manifest.json）にだけ存在する。これにより同じメタデータがセグメント数ぶん重複する事故を防ぐ。

#### 3.3.3 ファイル名規約

```
{通し番号5桁}_{表示名}.txt
```

- 通し番号は `00000` から。マニフェストの `segments` 配列の順序と一致
- 表示名は半角英数・日本語・アンダースコア・ハイフンのみ許可（OS 互換性のため）
- 表示名が長すぎる場合は先頭 40 文字で切る
- ファイル名は表示用でしかなく、**論理的な順序は manifest.json が決める**

並べ替え時はファイルをリネームする（通し番号を振り直す）。これによりファイルを Finder で見たときも順序が正しく見える。

### 3.4 管理領域 `.nexus-project/`

#### 3.4.1 version.json

```json
{
  "formatVersion": 1,
  "schemaVersion": 1,
  "installId": "01936b9c-...",
  "createdBy": "NEXUS 4.7.2"
}
```

この NEXUS プロジェクトがどのフォーマットバージョンで作られたか。マイグレーション時の判定に使う。

#### 3.4.2 journal.log

すべての破壊的操作（分割・統合・リネーム・削除・マニフェスト更新）を JSONL 形式で追記記録する。復旧・デバッグ・監査用。

```jsonl
{"ts":"2026-04-19T15:23:11+09:00","op":"segment.write","workId":"...","segmentId":"s-00001","checksumBefore":"sha256:...","checksumAfter":"sha256:...","charCountBefore":42179,"charCountAfter":42180}
{"ts":"2026-04-19T15:24:05+09:00","op":"segment.split","workId":"...","parentSegmentId":"s-00001","newSegmentIds":["s-00001","s-00003"],"splitAtChar":30000,"trigger":"auto"}
```

journal はローテートする（100 MB または 30 日で）。古いジャーナルは `.nexus-project/journal.archive/YYYY-MM.log.gz` に gzip 圧縮で保管。

#### 3.4.3 snapshots/

スナップショット機構は設計書 B で詳細化する。ここでは配置のみ：

```
.nexus-project/snapshots/
├── immediate/     ← 編集毎、直近10版、軽量
│   └── <workId>/
│       └── <timestamp>/
├── hourly/        ← 1時間毎、24時間保持
├── daily/         ← 1日毎、7日保持
└── pinned/        ← 手動マーク、無期限
```

作品固有のスナップショット（`ノチユ.nexus/snapshots/`）もあるが、プロジェクト全体の横断スナップショットは `.nexus-project/snapshots/` に集約される。冗長だが、どちらか片方が壊れても復旧できる設計。

### 3.5 連結ビューと仮想オフセット

ユーザーが見ている「全文」は、セグメントファイルをマニフェスト順に連結した仮想文字列。これを **連結ビュー** と呼ぶ。

連結ビュー内の位置（文字オフセット）と、セグメント内の位置（セグメント ID + オフセット）は相互変換可能：

```
連結オフセット 125000 
  = セグメント s-00001 内のオフセット 82820
  （s-00000 の文字数が 42180 だから 125000 - 42180 = 82820）
```

この変換テーブルは **セグメントマップ** と呼び、マニフェストの `segments[].charCount` から計算される。メモリに常駐する軽量テーブル（セグメント数ぶんのレコード）。

セグメントの内容が変わると charCount も変わるので、セグメントマップは編集のたびに更新される。全文検索・ジャンプ・エクスポート・プレビューはすべてセグメントマップ経由で連結ビュー座標を扱う。

### 3.6 マイグレーション経路

#### 3.6.1 単一 `.txt` → `.nexus`（新形式へ）

ユーザーが「この作品を分割モードにする」を選ぶと：

1. 元ファイル `作品名.txt` をスナップショットに保存
2. 新ディレクトリ `作品名.nexus/` を作成
3. セグメント境界を検出（F-11 の優先順位）
4. `segments/` 配下にセグメントファイルを書き出し
5. `manifest.json` を書き出し
6. 元の `作品名.txt` を `.nexus-project/migrated/作品名_YYYYMMDD.txt` に **移動**（削除しない）
7. NEXUS 上では新ディレクトリを開く

**元ファイルは削除されない**。マイグレーション後も元の単一ファイル形式が残る。ユーザーが気に入らなければ元ファイルに戻せる。

#### 3.6.2 `.nexus` → 単一 `.txt`（旧形式へ）

「この作品を単一ファイルに戻す」を選ぶと：

1. 全セグメントを順に連結（`segments[i].file` を charCount 順に読み込んで結合）
2. メタデータブロック `[METADATA]...[/METADATA]` を末尾に付与
3. `作品名.txt` として書き出し
4. `.nexus/` ディレクトリを `.nexus-project/unmigrated/作品名_YYYYMMDD.nexus/` に **移動**（削除しない）
5. スナップショットから復元できるよう 30 日保持

ここでもデータは消えない。

### 3.7 互換性の契約

#### 3.7.1 旧 NEXUS で新フォーマット作品を開いた場合

未対応バージョンの NEXUS が `.nexus` ディレクトリを開くと、`manifest.json` の `formatVersion` を見て未知のバージョンなら **読み取り専用モード**で全セグメントを結合表示する。編集はさせない。これにより：

- 古い NEXUS で誤って編集して壊す事故がない
- それでも作品は読める

#### 3.7.2 Finder で手動編集された場合

ユーザーが NEXUS を閉じたあと、Finder で `segments/` 内のファイルを直接編集した場合：

- 次回 NEXUS 起動時、`manifest.json` の checksum と実ファイルが食い違う
- NEXUS は変更を検出し、「外部から変更が検出されました」ダイアログを出す
- 選択肢：①マニフェストを更新して変更を受け入れる ②スナップショットから復元

どちらを選んでも、その時点のスナップショットが自動保存される。

### 3.8 セグメント境界と文字列の保全

境界が踏んではいけないもの（契約 C-6）：

- ルビ `《》` の内部
- 青空文庫タグ `［＃…］` の内部
- フォントタグ `{font:…}...{/font}` の内部
- `**強調**` の内部
- リンク `[[…]]`・`［［…］］` の内部

境界検出アルゴリズム（F-11）はこれらの**開いているペア構造**を追跡しながら境界候補を評価し、ペアの内部に境界が入る場合はその候補を却下する。

境界が確定したら、分割の直前に以下を確認する：

1. 境界直前と直後のセグメントそれぞれを単独でパースし、未閉合タグがないこと
2. 境界の両側を結合し直した結果が元と同じ（トラップ検出）

どちらかが失敗したら分割を中止し、journal にエラーを記録する。この状態でも元の単一セグメントは維持されるため、作品は壊れない。

## 4. この設計で守られること・守られないこと

### 守られること

- ユーザーから見て作品は 1 つの長編として振る舞う
- 自動分割が透明に行われ、打鍵応答は長編でも即時
- 既存の単一 `.txt` 作品は触らずに済む
- セグメントファイル単体を他エディタで開いても編集できる（緊急時の逃げ道）
- ルビ・青空文庫タグ・フォントタグは分割で壊れない
- マニフェストが壊れてもセグメント結合で作品を復元できる
- セグメントが壊れても他セグメントは無事

### まだ守られていないこと（後続の設計書で扱う）

- **書き込み中のクラッシュ時、部分書き込みファイルが残らない保証** → 設計書 B（atomic write）
- **複数インスタンスで同じ作品を開いた場合の編集衝突解決** → 設計書 B
- **30 日スナップショットの容量爆発を防ぐ仕組み** → 設計書 B
- **5 日分飛んだ事故を二度と起こさない具体メカニズム** → 設計書 B
- **章タブモードと連続エディタモードの実装詳細** → 設計書 C
- **いつ実装し、誰が書くか** → 設計書 D
