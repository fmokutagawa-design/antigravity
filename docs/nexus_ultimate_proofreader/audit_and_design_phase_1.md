# フェーズ 1 監査・設計報告書：スケーラブル校正エンジン

## 1. 現状の監査結果
- **ファイル**: `nexus_backend/proofreader.py`
- **問題点**: 
    - 線形検索モデル: ルールが増えるほど比例して遅くなる。
    - メモリ消費: 数千の正規表現オブジェクトを常にメモリに保持するのは、Windows 環境（特にメモリ 8GB 以下の環境）でリスクがある。
    - 拡張性: ルールが JSON ファイル一つに固まっており、特定のルール（例：時代小説用、現代劇用）の切り替えが困難。

## 2. 影響範囲の特定（絶対に潰さない機能）
1.  **会話文の保護**: `split_dialogue_and_narration` による保護ロジックは、新エンジンでも最優先で適用する。
2.  **ホワイトリスト**: ユーザーが設定した `nexus_whitelist.json` が、新ルールの大量導入によって無視されないことを保証する。
3.  **XML 互換性**: フロントエンドの `AuditReportWindow.jsx` が期待する `<correction>` タグの形式を維持する。

## 3. 新エンジン（V2）の設計案

### 3.1 階層型スキャンアーキテクチャ
ルールを 3 つの階層に分離し、実行順序を制御する：
- **Layer A (Fast)**: 固定文字列の置換（「ことができる」→「できる」等）。正規表現ではなく辞書引きによる高速処理。
- **Layer B (Regex)**: 複雑なパターン（「[ぁ-ん]{1,2}」の連続など）。最適化された正規表現エンジンを使用。
- **Layer C (Context)**: 章番号や時系列を考慮した物語監査。

### 3.2 高速化アルゴリズム
- **プレフィルタリング**: 本文中にそのルールの「キーワード」が含まれていない場合、高コストな正規表現マッチングをスキップする。
- **コンパイル済みキャッシュ**: ルールのパース結果をバイナリ形式でキャッシュし、サーバー起動時間を短縮。

## 4. バグ防止・安全策
- **ユニットテストの自動化**: 新旧エンジンで同じテキストを入力し、出力（校正結果）が一致するかを検証するスクリプトを同時作成。
- **リソース制限**: 一度の校正で消費する最大 CPU 時間とメモリに上限を設け、OS ごとフリーズするのを防ぐ。
- **Windows パス互換**: `os.path.join` を徹底し、絶対パスの混入をゼロにする（指示書 2/8 の成果を継承）。
