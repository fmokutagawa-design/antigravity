# ポストモーテム報告書：機能消失（デグレ）の発生と再発防止について

## 1. 事象の概要
2026年4月1日、リーダーモード（ReaderView）の復元作業において、直前に実装されていた「画像ドラッグ＆ドロップ機能」および「パフォーマンスパッチ D」が一時的に消失（デグレ）する事象が発生しました。

## 2. 原因分析
### 2.1 盲目的な過去の復元
本来のリーダーモードが動作していた過去のコミット (`ba7449`) からコードを復元する際、その後に積み上げられた改善（画像D&D等）が `App.jsx` に含まれていないことに気づかず、古いベースラインのコードでファイルを上書きしてしまいました。

### 2.2 暗黙の共有知の欠如
複数のAIエージェントが同一の中核ファイル（特に `App.jsx`）を短期間に直接書き換えていたため、エージェント間での「直近の大きな変更」に関する情報共有が不十分でした。

### 2.3 確認不足
復元作業が「指定された機能（リーダーモード）の復活」のみに集中してしまい、既存機能のデグレーションチェック（リグレッションテスト）をマージ前後に行わなかったことが直接的な原因です。

## 3. 再発防止策（AI開発ガイドライン）
今後、AIエージェントおよび開発者は以下の「仕組み」を徹底します。

### 3.1 変更前の `git diff` による現状把握
中核ファイルを変更する前に、必ず `git diff` や `git log` を確認し、自分が今から上書きしようとしているコードに「自分が知らない新しいロジック」が含まれていないかを精査します。

### 3.2 `git cherry-pick` / 手動パッチの優先
ファイルを丸ごと過去の状態で上書き（`git restore` 等）することは極力避け、必要な機能だけを抽出して現在のコードベースに「移植」する手法を原則とします。

### 3.3 結合確認（ハイブリッド検証）
機能の追加・変更後は、新機能だけでなく、以下の「主要機能」が壊れていないか、ブラウザツール等を用いてクロスチェックを行います。
- 画像D&Dおよび挿絵記法のレンダリング
- ハイライト制限等のパフォーマンス最適化
- プレビューとリーダーモードの共存
- 起動時のコンソールエラー（初期化順序エラー等）の有無

### 3.4 ブランチおよびコミットの最小化
大きなリファクタリングや機能復旧を行う際は、必ず専用のブランチを作成し、`main` への統合は差分を慎重にレビューした上で行います。

## 4. 結末
本件により一時的な混乱を招きましたが、現在はすべての機能が「真の統合版」として `main` ブランチに反映され、動作することが確認されています。本レポートをプロジェクトの「記憶」とし、今後の開発の教訓とします。
