# 設計書 C：UI 設計

NEXUS 透明分割機構の第 3 部。ユーザーが実際に触れる操作を定義する。

## 0. 設計原則

### 0.1 二つの顔、一つの作品

ユーザーは同じ作品を **連続エディタモード** と **章タブモード** のどちらでも開ける。どちらで触っても作品は破壊されず、内容は同一。モードは「見え方」の差であって、ファイル構造を変えるものではない。

### 0.2 「分割されている」ことを見せすぎない

NEXUS が裏で分割していることは、原則ユーザーから見えない。境界は連続エディタモード上では **薄い水平線とオプションの章タイトル** として控えめに存在する。章タブモードでは明示的に分離するが、これは「章を切り替えて書きたい」という意図を持つユーザーの選択。

### 0.3 安全な操作、危険な操作

**安全な操作**（ユーザー確認不要）：カーソル移動、編集、セグメントまたぎ Undo/Redo、検索、表示モード切替。

**不可逆操作**（必ずユーザー確認ダイアログを経由）：セグメントの分割、統合、並べ替え、削除、マイグレーション（単一 `.txt` ↔ `.nexus`）、スナップショットからの復元。

### 0.4 設計書 B からの反映

- 自動境界提案はユーザー承認経由（F-11.5）
- immediate スナップショットは 10 分毎 & idle 時のみ発動
- V-3a（50% 短縮）は静かに追加スナップショット、V-3b（95% 短縮）のみダイアログ
- ロック衝突時のデフォルトは「読み取り専用」
- pinned スナップショットは作品内冗長配置を実装

## 1. 連続エディタモード

### 1.1 基本レイアウト

現行の NEXUS エディタ画面をそのまま踏襲する。違いは中身だけ：

```
┌─────────────────────────────────────────────────────────────────┐
│ [左パネル]  │                                                    │
│ ファイル    │                                                    │
│ ツリー      │         縦書きエディタ                              │
│ リファレ    │         （全セグメント連結表示）                     │
│ ンス        │                                                    │
│ ...         │                                                    │
│             │                                                    │
│             │                                                    │
│ [表示]      │                                                    │
│ ○連続エディタ│                                                    │
│ ○章タブ     │                                                    │
│             │                                                    │
│             │                                                    │
└─────────────────────────────────────────────────────────────────┘
```

左パネルに「表示モード」のトグルを追加。作品ごとに記憶される。

### 1.2 セグメント境界の見せ方

連続エディタモードでは、セグメント境界は以下のように表示する：

```
...そして戦いは終わった。
それは彼が望んだ結末ではなかったが、もう
後戻りはできなかった。

              ━━━━━━━━━━━━━━━━━━━━━━
              ◆ 第二章　嵐の前触れ
              ━━━━━━━━━━━━━━━━━━━━━━

雪がちらついていた。窓の外は......
```

- **水平線**（縦書きの場合は垂直線）で境界を示す。色は本文色の 20% 程度の淡さ
- **セグメント表示名**（manifest.json の displayName）をその境界に小さく表示
- 表示名がない場合は「◆ ここで区切り」とだけ表示
- ユーザー設定で **境界を完全に非表示**にもできる（執筆中は何も見せたくない派向け）

### 1.3 セグメント境界の操作

境界部分は **クリック可能なソフトターゲット**。以下の操作を提供：

- **クリック** → ポップオーバーで「章タイトルを編集」「境界を削除（隣と統合）」「この章で区切り直す」
- **ダブルクリック** → 章タブモードに切り替わり、この章にフォーカス
- **右クリック** → 境界操作のフルメニュー

### 1.4 カーソル挙動：境界ゼロ摩擦

境界を跨ぐカーソル操作で違和感が出ないようにする：

| 操作 | 挙動 |
|---|---|
| ← / → / ↑ / ↓ でセグメント境界を越える | 次セグメント先頭（または末尾）に連続して移動 |
| セグメント境界に **Backspace** | 前セグメントの末尾文字を削除。結果としてセグメントが統合される可能性あり（§5.2） |
| セグメント境界に **Delete** | 次セグメントの先頭文字を削除 |
| **Shift+矢印**で境界を越えた選択 | 複数セグメントを跨ぐ選択範囲 |
| **Cmd+A** 全選択 | 作品全体 |
| **Cmd+← / Cmd+→** | 現在の段落の先頭/末尾（境界は越えない。段落単位の移動のほうが意図に合う） |
| **Cmd+↑ / Cmd+↓** | 作品全体の先頭/末尾（境界を越える） |

### 1.5 仮想化（Phase 2 の重要最適化）

連続エディタモードで 42 万字 + N セグメントを全部 DOM マウントしたら、元の問題に逆戻りする。以下の仮想化を行う：

- **画面表示中 + 前後 1 セグメント** のみ実 DOM を持つ
- それ以外のセグメントは、プレースホルダ `<div style="height: 〇〇px">` で高さだけ確保
- スクロール位置の変化を IntersectionObserver で検知し、前後のセグメントをマウント・アンマウント
- マウント済みセグメントの文字数上限は合計 10 万字程度に抑える

詳細実装は設計書 D の Phase 2 で扱う。

### 1.6 IME（日本語変換）挙動

- IME 変換中はセグメント境界をまたいだ textarea 更新を禁止（現行の `isComposingRef` ガードを流用）
- 変換確定後のカーソル位置が別セグメントに入っていたら、そのセグメントにフォーカス移動
- 変換候補ウィンドウが境界で切られないよう、ポップアップ位置の再計算を行う

### 1.7 スクロール

- 縦書きなら水平スクロール
- 各セグメントはそれぞれが独立した `<div>` コンテナだが、親で統一的にスクロール
- Cmd+G などで検索結果にジャンプすると、該当セグメントを即マウントして滑らかスクロール

## 2. 章タブモード

### 2.1 レイアウト

```
┌─────────────────────────────────────────────────────────────────┐
│ [左パネル]  │  [プロローグ] [第1章] [第2章 ◉] [第3章] [＋新規章] │
│             │  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━  │
│             │                                                    │
│             │         縦書きエディタ                              │
│             │         （選択された章のみ）                         │
│             │                                                    │
│             │                                                    │
└─────────────────────────────────────────────────────────────────┘
```

タブバーが上部にある。アクティブなタブのセグメントだけエディタに表示される。

### 2.2 タブ操作

- **タブクリック** → そのセグメントに切替（書きかけは autoSave でディスクに確定してから切替）
- **タブ右クリック** → メニュー（リネーム／削除／前の章と統合／次の章と統合／前に挿入／後に挿入）
- **タブドラッグ** → 順序入れ替え（manifest を atomic update）
- **＋新規章** → 末尾に空セグメント追加、リネームプロンプト
- **タブの文字数表示**：タブ名の下に小さく `42,180字`

### 2.3 章タブモード特有の挙動

- 章をまたぐコピペ：別タブから貼り付けしても問題なく動く（テキストは単なる文字列として扱うため）
- 検索：章タブモードでも全文検索はデフォルトで全章対象。ヒットが別章にあればタブが点滅する

### 2.4 タブが多すぎるとき

13 章以上になるとタブが画面に収まらない。以下の仕組み：

- タブバーは水平スクロール可能
- **章一覧ドロップダウン**（右端の `▼` ボタン）で全章のリスト表示
- **Cmd+[ / Cmd+]** で前後の章に移動
- **Cmd+数字** で n 番目の章へジャンプ

## 3. モード切替

### 3.1 切替 UI

左パネルに以下のトグル：

```
表示モード
┌────────────────────┐
│ ○ 連続エディタ      │
│ ● 章タブ            │
└────────────────────┘
☐ 境界を非表示にする （連続モード時のみ有効）
☐ 章タイトルを表示   （連続モード時のみ有効）
```

### 3.2 切替時の挙動

- 即時切替。現在の編集内容は autoSave で確定してから切り替える
- 連続エディタモードでカーソルがあった位置 → 章タブモードで対応する章にフォーカス
- 章タブモードで編集中の章 → 連続エディタモードでその位置にスクロール

### 3.3 設定の永続化

作品ごとに manifest.json の `uiPreferences.defaultMode` に記録される。次回その作品を開いたときに同じモードで開く。

## 4. 境界提案 UI（F-11.5 + インライン提案）

### 4.1 インライン境界ヒント（冬之助さん要望）

ユーザーが打鍵中、行頭に以下のパターンを入力した瞬間にヒントが出る：

- `■` + 何か
- `第[数字]章`、`第[数字]話`、`第[数字]幕`
- `# ` から始まる（Markdown 見出し）
- `［＃大見出し］` `［＃中見出し］`（青空文庫タグ）

#### 4.1.1 ヒントの見た目

```
                                   ↓このアイコンが行頭の左に小さく出る
  ■ 第二章　嵐の前触れ     [◆]
  
  雪がちらついていた。
```

- アイコンは行の左側（横書き時）または下側（縦書き時）に控えめに配置
- 色は本文色の 30% 程度、hover で 100%
- **打鍵を邪魔しない**：クリックされるまで何もしない。時間が経てば薄くフェードアウト

#### 4.1.2 クリック後の挙動

アイコンをクリックすると、**境界確認ポップオーバー**が開く：

```
┌──────────────────────────────────────────┐
│ ここで章を区切りますか？                  │
│                                          │
│ ＜前の内容（末尾400字）＞                  │
│ ...そして戦いは終わった。                 │
│ それは彼が望んだ結末ではなかった......    │
│                                          │
│ ＜ここで区切り＞                          │
│                                          │
│ ＜後の内容（先頭400字）＞                  │
│ ■ 第二章　嵐の前触れ                     │
│ 雪がちらついていた。窓の外は......        │
│                                          │
│ 章タイトル: [第二章　嵐の前触れ      ]    │
│                                          │
│ [区切る]  [区切らない]                    │
└──────────────────────────────────────────┘
```

- 章タイトルは自動抽出（`■ 第二章　嵐の前触れ` から `第二章　嵐の前触れ` が抽出される）
- ユーザーが編集可能
- 「区切る」で分割実行。スナップショットが先に取られる
- 「区切らない」でヒントが消える（この位置のパターンは**学習され、同じパターンで二度とヒントを出さない**）

### 4.2 自動分割提案ダイアログ（閾値超過時）

1 セグメントが 6 万字を超えると、システムが idle 時に境界候補を探す。見つけたら以下のダイアログ：

```
┌──────────────────────────────────────────┐
│ 作品が大きくなってきました                │
│                                          │
│ 現在、以下の章が 6 万字を超えています：    │
│   第1章 オデッサの残光（64,200字）         │
│                                          │
│ 以下の位置で区切ることを提案します：       │
│                                          │
│ ＜前＞                                   │
│ ...アムロはハッと息を呑んだ。             │
│ それは彼の人生を変える一瞬だった。         │
│                                          │
│ ＜後＞                                   │
│ ■ 一週間後                               │
│ 少佐は窓際に立っていた......              │
│                                          │
│ 章タイトル候補: [一週間後              ]  │
│                                          │
│ [この位置で区切る] [別の位置を提案]        │
│ [今は区切らない]   [今後これを提案しない]   │
└──────────────────────────────────────────┘
```

**4 つのボタンの意味**：

- **この位置で区切る**：分割実行、スナップショット取得
- **別の位置を提案**：同セグメント内の他の境界候補を表示（もしあれば）
- **今は区切らない**：今回だけスキップ。後でまた提案される（6 万字のまま続くと次の idle で再提案）
- **今後これを提案しない**：このセグメントに対して永続的に提案を止める（8 万字、10 万字になっても勝手に出さない）

### 4.3 境界候補が見つからない場合

6 万字を超えたが明確な境界が見つからない場合：

```
┌──────────────────────────────────────────┐
│ 章の区切りが見つかりません                 │
│                                          │
│ 第1章 オデッサの残光（74,000字）が        │
│ 大きくなっていますが、章の区切り記号       │
│ （■ や 第◯章 など）が見つかりません。     │
│                                          │
│ このままでも問題ありませんが、             │
│ 非常に大きくなると打鍵が遅くなる可能性が   │
│ あります（目安: 10 万字超）。              │
│                                          │
│ [今のままにする]                          │
│ [手動で区切り位置を決める]                 │
│ [文字数で機械的に区切る（推奨しない）]     │
└──────────────────────────────────────────┘
```

ユーザーの意図なしに機械的分割は行わない。

### 4.4 手動分割

連続エディタモード：**右クリック → 「ここで章を分ける」**。カーソル位置で分割。

```
右クリックメニュー：
  ルビ挿入
  リンク挿入
  フォント変更 ▸
  ──────────
  ここで章を分ける           ← これを追加
  ──────────
  切り取り
  コピー
  貼り付け
```

クリック後、境界確認ポップオーバー（§4.1.2 と同じ）が開く。

章タブモード：**カーソル位置で右クリック → 「ここで次の章に分ける」**、または **タブ右クリック → 「カーソル位置で分割」**。

### 4.5 手動統合

章タブモード：**タブ右クリック → 「前の章と統合」「次の章と統合」**。

連続エディタモード：**境界の線をクリック → ポップオーバーで「境界を削除（統合）」**。

統合前に確認ダイアログ：

```
「第2章 と 第3章 を統合しますか？
 統合後の文字数: 58,400字
 
 統合前の状態はスナップショットに保存されます。
 後から分割しなおすこともできます。
 
 [統合する]  [キャンセル]」
```

## 5. 編集操作の境界またぎ

### 5.1 選択範囲が境界をまたぐ

複数セグメントを選択してコピー・切り取り・削除する操作は、内部的には各セグメントを編集してマニフェストを更新する複合操作。原子的に扱う（トランザクション化、設計書 B §2.4）。

- **Cmd+C**（コピー）：選択範囲のプレーンテキストをクリップボードへ。境界の薄い線は出力されない
- **Cmd+X**（切り取り）：上記 + 該当範囲を削除。結果として空になったセグメントは **削除候補**となり、次の idle 時に統合提案
- **Delete キーでの削除**：上と同じ

### 5.2 Backspace が境界を跨ぐとき

境界の直後で Backspace を押すと、前セグメントの末尾文字を削除する。これは「境界を削除する」操作ではなく、「境界直前の文字を削除する」操作。

境界自体を削除したい場合は §4.5 の手動統合で行う。これを意図的に分離するのは、Backspace 連打での事故を防ぐため。

### 5.3 Undo/Redo

Undo 履歴は **作品単位**で管理する。境界操作（分割・統合）も Undo 可能。

```
Undo 可能な操作：
  - 通常の文字入力・削除
  - 境界追加（分割）
  - 境界削除（統合）
  - 章タイトル変更
  - 章の並び替え
  - セグメント削除

Undo 不可能な操作（スナップショット復元でのみ戻せる）：
  - マイグレーション（単一 .txt ↔ .nexus）
  - 古い版のスナップショットの削除（_trash から 30 日経過後）
```

Undo の実装は、各セグメント単位の履歴ではなく、**作品全体のマニフェスト＋セグメントの組**を履歴エントリとして持つ。メモリ効率は悪化するが、境界操作を含む Undo を正しく扱うためにはこれが必要。

### 5.4 ドラッグ&ドロップ

- 画像の D&D：現行挙動を踏襲、`［＃挿絵（filename）入る］` タグを挿入
- テキストの D&D：境界をまたいだ範囲でも動作、内部的には切り取り + 貼り付け

## 6. 全文検索・置換・ジャンプ

### 6.1 全文検索

**Cmd+F**（または左パネルの検索アイコン）：

```
┌──────────────────────────────────────────┐
│ 🔍 検索: [アムロ                    ]     │
│                                          │
│ 全 42 件                                 │
│                                          │
│ ☐ 大文字小文字区別                        │
│ ☐ 正規表現                                │
│ ☑ 全章を検索                              │
│                                          │
│ ── 結果 ──                               │
│ 第1章 オデッサの残光                      │
│   12行目: アムロは空を見上げた...         │
│   89行目: 「アムロ、いい加減にしろ」      │
│   ...                                    │
│                                          │
│ 第2章 嵐の前触れ                          │
│   3行目: アムロの目は赤く...             │
│   ...                                    │
└──────────────────────────────────────────┘
```

**仕様**：
- 各セグメントを並列に検索（Worker で）
- 結果は章ごとにグルーピング
- クリックでジャンプ（連続エディタモードならスクロール、章タブモードならタブ切替）
- ジャンプ先のセグメントが未マウントなら即座にマウント
- **章タブモードで別章のヒットをクリックすると自動的に章切替**

### 6.2 全文置換

**Cmd+Shift+F** または検索パネルから「置換」モードへ：

```
┌──────────────────────────────────────────┐
│ 🔍 検索:   [アムロ                  ]     │
│ 置換:     [アムロ・レイ             ]     │
│                                          │
│ 全 42 件                                 │
│                                          │
│ [次を置換]  [すべて置換（42件）]          │
└──────────────────────────────────────────┘
```

**「すべて置換」実行時**：

1. 事前に手動スナップショットが作成される（operation: "mass-replace"）
2. 全セグメントで並列に置換
3. 各セグメントを atomic write
4. 完了後、置換件数を表示
5. **「置換を取り消す」ボタン**が 60 秒間表示される（クリックでスナップショットから即復元）

### 6.3 ジャンプ

- **Cmd+G**: 次の検索結果
- **Cmd+Shift+G**: 前の検索結果
- **ファイル行番号ジャンプ**（Cmd+L）: 作品内の行番号（全セグメント横断）で指定
- **章ジャンプ**（Cmd+J）: 章一覧からクリックでジャンプ

## 7. プレビュー・リーダー連結表示

### 7.1 プレビュー

現行の NEXUS プレビュー機能と同じ操作だが、**常に全章連結**で表示される。ページネーション、ルビ表示、フォント切替、背景色などは現行踏襲。

- セグメント境界は **自然な章切替**として表示（水平線 + 章タイトル）
- 賞応募用のページ数カウントもこの連結表示ベースで計算

### 7.2 リーダー

現行のリーダーモードと同じ。縦書きで横スクロール、見出しジャンプ、フォント・サイズ・カラー切替。全章連結。

**リーダーモード特有の最適化**：

- 連続エディタモードと違い、**編集しない**ので仮想化レベルを上げても UX に影響しない
- 現在ページ ± 5 章のみマウント、それ以外はプレースホルダ
- フォントサイズ変更時は全マウントセグメントを再計算

## 8. エクスポート

### 8.1 エクスポート UI

ファイル → エクスポートメニュー：

```
エクスポート
  プレーンテキスト（全文結合 .txt）  ← 賞応募用
  PDF（縦書き、原稿用紙）
  PDF（縦書き、無地）
  PDF（横書き）
  EPUB
  ────────────────
  現在の章のみ ▸
  章を選んで ▸
  ────────────────
  設定...
```

### 8.2 全文結合 .txt の仕様

**賞応募用の最重要エクスポート**。絶対に壊さない。

- 全セグメントを `segments[].file` の順に読み込み
- 各セグメント間に **設定で指定された区切り**を挿入（デフォルト: 空行 2 つ）
- 末尾にメタデータブロック（既存仕様）を追加するかどうかは設定
- エクスポート前に**自動で pinned スナップショット**を作成（`submission_YYYYMMDD_HHmm`）

```
出力例（区切り = 空行 2 つ、メタデータなし）:

[プロローグの内容]


[第1章の内容]


[第2章の内容]
...
```

### 8.3 PDF / EPUB

- 全セグメント連結して入力とする
- フォント・レイアウト・ルビ処理は既存エクスポート経路を踏襲
- 章の改ページは manifest の境界を使う
- 目次は manifest の `displayName` から自動生成

### 8.4 部分エクスポート

- **現在の章のみ**：章タブモードでアクティブな章のみ、または連続エディタモードでカーソルがある章
- **章を選んで**：複数選択可能なチェックリスト UI

部分エクスポートは賞応募の下書き用（分冊チェックなど）に使われる想定。


- デフォルトフォーカスは「読み取り専用で開く」

- タイトルバーに「読み取り専用」の赤いバッジ

起動時または再フォーカス時に、mtime + checksum の食い違いを検出したとき：

```

│ ⚠️ 外部から変更されたファイルがあります     │
│                                          │
│ segments/00003_第3章.txt                 │
│   マニフェスト上: 2026-04-19 10:00        │
│   実ファイル:    2026-04-19 14:30         │
│   文字数:       42,180 → 41,500 (-680)    │
│                                          │
│ 対処を選んでください：                     │
│                                          │
│ [外部変更を受け入れる]                    │
│   現在のファイル状態を正とします           │
│   （NEXUS 内の最新スナップショットは保持）  │
│                                          │
│ [マニフェストに合わせて復元する]          │
│   スナップショットから復元します           │
│   （ファイル上の変更はスナップショットへ）  │
│                                          │
│ [後で決める（読み取り専用で開く）]         │
└──────────────────────────────────────────┘
```

どの選択肢でも、実行前に**両方の状態**がスナップショットに保存される。後から取り消せる。

2. **画面が idle**（マウス動きなし、キー入力なし 30 秒以上）で初めて提案ダイアログが出る
3. それでも作業再開（キー入力）を検知したら即座にダイアログを閉じ、**1 時間後に再試行**

執筆の邪魔を絶対にしない。

## 16. この設計で守られること・守られないこと

### 守られること

- ユーザーは境界を意識せずに長編を執筆できる
- 境界ヒントは邪魔にならず、分割は必ずユーザー承認を経る
- 2 つの表示モードで執筆スタイルに合わせられる
- 全文検索・置換・ジャンプが境界をまたいで動く
- プレビュー・エクスポートは全章連結で出る
- スナップショットからの復元は常にユーザーに選択肢を提示する
- 競合・外部改変・破損は検出され、ユーザーに選択肢が提示される
- 不可逆操作（分割・統合・マイグレーション・置換・復元）は必ず事前スナップショット

### まだ守られていないこと（後続の設計書で扱う）

- **いつ、誰が、どの順序で実装するか**：設計書 D
- **既存 NEXUS コードのどこを壊し、どこを温存するか**：設計書 D
- **モバイル・iPad 完全対応**：将来の別企画
- **複数ユーザー同時編集**：このシステムでは非対応。そもそも NEXUS は個人ツール

### 今はやらないこと（Phase 2 以降）

- 差分スナップショット（圧縮効率向上）
- 版の分岐管理（Git 的な branch）
- 章ごとのコメント機能
- 複数 PC 間の同期

- 手動分割（右クリックから）

- プレビュー連結表示
