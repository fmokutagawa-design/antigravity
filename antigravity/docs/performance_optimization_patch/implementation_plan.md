# 実装計画：パフォーマンス最適化パッチの適用

## 概要
ユーザーから提供されたパフォーマンス最適化パッチを適用します。このパッチは、座標計算の二重加算バグの修正、段落単位のメモ化による高速化、および Web Worker 応答のデバウンス処理を含んでいます。

## 現状の課題とパッチの狙い
1. **座標計算の二重加算**: Worker と Synthesis 両方で `lineOffset` を足していた問題を修正。
2. **キャッシュの揮発性**: `lineOffset` が全項目に含まれていたため、先頭に1行入るだけで全段落のキャッシュが無効化されていた。ローカル座標（line=0開始）でキャッシュすることで、再利用性を高める。
3. **合成処理の負荷**: 14万字クラスでは合成処理だけで 70ms 程度かかっていたものを、段落単位のメモ化（UTF-16長計算の省略）とデバウンス（40ms）により、タイピングのラグを劇的に改善する。

## 変更内容

### 1. `src/utils/positionWorker.js` [MODIFY]
- `para_positions` 処理において `lineOffset` の加算を削除。
- 段落内のローカル座標（line=0始まり）をそのまま返すように変更。

### 2. `src/components/Editor.jsx` [MODIFY]

#### 2.1. 段落メモの導入
- `composedSegCacheRef` を追加し、段落ごとの `charArray`, `utf16Lens`, `totalLines` をメモ化。これにより、座標の全文合成時に `codePointAt` などの重い処理を省略。

#### 2.2. Worker 送信ロジックの簡略化
- 送信前に `offsetMap` を構築する処理を削除（もう不要なため）。
- 依存配列を `[baseMetrics.maxPerLine, debouncedDocument]` に戻す。
  - > [!IMPORTANT]
  > 前回の修正で「Tick カウンター」を導入しましたが、パッチの設計（デバウンス入り）がこれを代替・改善する形になっているため、パッチの設計に従って直接依存に戻します。ただし、「Hook サイズ変更エラー」が再発しないか慎重に確認します。

#### 2.3. 合成処理のデバウンスと最適化
- 合成処理全体を `setTimeout(run, 40)` で包み、Worker からのバッチ応答が連続しても、最終的に1回だけ合成が走るように変更。
- 合成ループ内で累積的な `lineOffset` を各座標に加算。
- 依存配列を `[paraPosCache, debouncedDocument, baseMetrics.maxPerLine]` に戻す。

## 懸念点
- **Hook サイズ変更エラー**: `debouncedDocument` 配列そのものを `useEffect` の依存配列に入れると、環境によってはエラーが再発する可能性があります。
  - **対策**: もしエラーが出る場合は、パッチのロジックを維持しつつ、依存関係だけを Tick カウンターに戻すハイブリッド方式を採用します。まずはパッチ通りに適用を試みます。

## 検証計画
- 10万字以上のテキストを貼り付け、スクロールや選択、タイピングに遅延がないか確認。
- 改行挿入時に、後続の段落が正常に再配置されるか（`lineOffset` の加算が正しく機能しているか）を確認。
- `git commit` および `push` の実行。
