# 設計書 B：安全性設計

NEXUS 透明分割機構の第 2 部。原稿を失わないことを機能ではなく契約として組み込む。

## 0. この設計書が答える問い

- どんなバグ・クラッシュ・電源断が起きても、原稿は消えないか？
- 過去の「5 日分飛んだ」ような事故は、どの具体メカニズムで防がれるか？
- 二重に走らせた NEXUS が原稿を殴り合わないか？
- スナップショットはディスクを食い潰さないか？
- 自動分割で境界計算をミスっても作品は無事か？

## 0.1 前提：A からの追加要件

設計書 A の「いいと思います」レビューを受けて、以下を契約として追加する：

- **F-11.5**: 自動分割の前に**ユーザー確認ダイアログ**を出す。境界候補の前後 400 字程度を並べて表示し、「ここで区切ってよいか」を聞く。承認された境界のみ分割が実行される
- **D-6**: セグメント境界は**ユーザーが少なくとも 1 度承認したもののみ自動適用**される。新規作品で未承認の場合は、境界提案 UI を経由してから確定する
- **F-11.6**: 承認された境界位置は**永続記憶**される（manifest に記録）。同じ作品で類似の境界候補が出た場合は優先的に採用される

この F-11.5 は単に UX の親切さだけでなく、「不可逆操作はユーザー承認を経る」という安全設計の柱となる。設計書 B ではこの柱を複数の保護層で支える。

## 1. 脅威モデル

### 1.1 原稿が失われる経路の全列挙

過去の事故と、理論上の失敗経路を全部洗い出す。

#### T-1. 書き込み中の異常終了

書き込み中にプロセスがクラッシュ、OS がフリーズ、電源断、macOS のスリープ／強制終了。`fs.writeFile` が途中で止まると、ファイルが半分だけ書かれた壊れた状態になる。

**過去事例**：2026 年 4 月の「5 日分飛んだ」事故がこれに該当する可能性が高い。パフォーマンス改善のコード変更中に書き込み経路でバグを踏み、途中書き込み状態が残ったまま次回起動時に上書きされた。

#### T-2. ロジックバグによる空文字列書き込み

`onChange` や `autoSave` のタイミングズレで、ファイル内容が一瞬 空文字列 `""` になった瞬間に保存が走る。既存コードには `if (!debouncedText || debouncedText.length === 0) return;` のガードが入っているが、これを迂回する経路が残っている可能性。

**過去事例**：NEXUS のバックグラウンドで controlled→uncontrolled 移行中に localText が undefined になる瞬間があった。そこで保存が走れば原稿消失。

#### T-3. ファイル切替と保存のレース

ファイル A を開いて編集中にファイル B に切替。切替時の onChange が A の内容を B に書き込む、または B の初期内容（空）が A に書き込まれる。

**過去事例**：fileId useEffect と value useEffect の実行順序、activeFileHandle の更新タイミング、autoSave の非同期完了タイミングの 4 者のレースで実装ミスが起きた履歴あり。

#### T-4. マニフェスト更新の中断

分割・統合・リネーム操作中に異常終了し、manifest.json は新形式、実ファイルは旧形式、みたいな食い違いが残る。

#### T-5. ファイルシステム障害

- exFAT の書き込み失敗（既知の NEXUS 問題、`docs/app_build_exfat_workaround/` 参照）
- iCloud Drive で同期中に NEXUS が上書き
- ネットワークドライブの切断
- ディスク容量不足で書き込みが途中で止まる

#### T-6. 複数インスタンスの同時編集

macOS の Dock から NEXUS を 2 つ起動、あるいは Claude in Chrome のような別チャンネルからの書き込みと衝突。後から保存したインスタンスが、先のインスタンスの編集を上書きする。

#### T-7. 境界計算ミスによるデータ欠落

分割時にルビや青空文庫タグの内部で境界を引いてしまい、片側のセグメントにタグの前半、もう片側に後半が残る。結合しなおしても壊れる。

#### T-8. 統合時の結合ミス

2 つのセグメントを結合する際、末尾の改行処理で 1 文字消える／2 文字増える。10 万回の編集で少しずつ原稿が壊れる。

#### T-9. スナップショットの破損伝播

スナップショット自体が書き込み中に壊れると、復旧経路が失われる。「スナップショットから戻したら原稿がもっと壊れた」は最悪の事故。

#### T-10. 外部からの改変

Finder で直接ファイルを消す、別エディタで開いて別形式で保存、Time Machine の復元で古いファイルに戻される。NEXUS が次回起動時にそれを「ユーザーの意図した変更」と誤認して checksum を更新してしまう。

#### T-11. バックアップの取り忘れ

ユーザーが定期バックアップを取っていない。iCloud・Time Machine・外付け HDD が全部失敗している状態で、NEXUS のスナップショットだけが頼り。スナップショットが無いと何もできない。

### 1.2 対策マトリクス

| 脅威 | 主対策 | 副対策 |
|---|---|---|
| T-1 書き込み中断 | **atomic write**（§2.1） | ジャーナル（§2.5）、スナップショット（§3） |
| T-2 空文字列書き込み | **多層バリデーション**（§2.2） | ジャーナル、スナップショット |
| T-3 切替レース | **世代カウンタ**（§2.3） | atomic write、スナップショット |
| T-4 マニフェスト中断 | **トランザクションログ**（§2.4） | マニフェスト二重化（§2.4.3） |
| T-5 FS 障害 | **書き込み成功確認**（§2.1） | ジャーナル、スナップショット |
| T-6 複数インスタンス | **ロックファイル**（§4） | 世代カウンタ、衝突検出 |
| T-7 境界計算ミス | **パース検証**（§5.1） | ユーザー承認 UI（F-11.5） |
| T-8 結合ミス | **ラウンドトリップ検証**（§5.2） | スナップショット |
| T-9 スナップショット破損 | **多世代化**（§3） | 冗長配置（§3.5） |
| T-10 外部改変 | **mtime + checksum 検証**（§6） | 競合ダイアログ |
| T-11 ユーザーバックアップなし | **スナップショット 30 日保持**（§3） | エクスポート機能（設計書 C） |

## 2. ファイル書き込み層

### 2.1 Atomic Write：すべての書き込みはこれを通す

NEXUS 内のあらゆる書き込みは、以下の手順を踏む。**この手順から外れた書き込み経路は存在してはならない**。

#### 2.1.1 手順

```
1. 書き込み対象: /path/to/segments/00001_xxx.txt
2. 書き込みペイロード P を準備
3. 事前検証（§2.2）が通ることを確認
4. 同ディレクトリに一時ファイルを作成: /path/to/segments/00001_xxx.txt.tmp.{pid}.{ts}
5. 一時ファイルに P を fs.writeFile で書く
6. 一時ファイルを fsync（OS バッファを物理ディスクへ flush）
7. 一時ファイルを fs.readFile で読み直し、チェックサムが P と一致することを確認
8. 一時ファイルを fs.rename で本来のパスへ置換
   （POSIX atomic rename により、このステップは原子的）
9. 親ディレクトリを fsync（rename を確定）
10. ジャーナルに操作を記録（§2.5）
```

異常終了がどの時点で起きても、**本来のパスのファイルは古い内容のまま**である。半端な書き込みが本番ファイルに触れることは構造的にありえない。

#### 2.1.2 Electron/Node.js 実装

```js
// utils/atomicWrite.js
import { promises as fs } from 'node:fs';
import { open } from 'node:fs/promises';
import path from 'node:path';
import crypto from 'node:crypto';

export async function atomicWriteFile(filePath, content) {
  const dir = path.dirname(filePath);
  const basename = path.basename(filePath);
  const tmpPath = path.join(
    dir,
    `${basename}.tmp.${process.pid}.${Date.now()}`
  );

  const expectedHash = sha256(content);
  let fh;

  try {
    // 1. 一時ファイルに書き込み
    fh = await open(tmpPath, 'w');
    await fh.writeFile(content, 'utf8');
    await fh.sync(); // fsync
    await fh.close();
    fh = null;

    // 2. 読み直して検証
    const readBack = await fs.readFile(tmpPath, 'utf8');
    if (sha256(readBack) !== expectedHash) {
      throw new Error('atomic write: readback checksum mismatch');
    }

    // 3. rename（POSIX atomic）
    await fs.rename(tmpPath, filePath);

    // 4. ディレクトリ fsync
    const dirFh = await open(dir, 'r');
    try { await dirFh.sync(); } finally { await dirFh.close(); }

    return expectedHash;
  } catch (err) {
    // 失敗時は一時ファイルを掃除（本来ファイルは触らない）
    if (fh) { try { await fh.close(); } catch {} }
    try { await fs.unlink(tmpPath); } catch {}
    throw err;
  }
}

function sha256(s) {
  return crypto.createHash('sha256').update(s, 'utf8').digest('hex');
}
```

#### 2.1.3 既存コードへの適用

現行 `fileSystem.electron.js` の `writeFile`・`writeFileBinary`・`createFile` はすべて `atomicWriteFile` を経由するよう書き換える。main プロセス側（`electron/main.cjs`）も同様。ブラウザ版（File System Access API）は OS レベルの atomic rename が保証されないため、別途対応（§2.6）。

#### 2.1.4 ディスクフル・書き込み失敗時

`fh.writeFile`・`fh.sync`・`fs.rename` のいずれが失敗しても、**本番ファイルは触れられていない**。失敗を呼び出し側に throw して、上位で「保存失敗」のトーストを表示。原稿は直前の保存済み状態のまま。

### 2.2 多層バリデーション：保存禁止条件

atomic write の**事前検証**で以下を全部通す。1 つでも失敗したら書き込みを中止する。

#### 2.2.1 ペイロード検証

```js
function validatePayload(content, context) {
  // V-1: 空文字列禁止（空の作品を明示的に作る新規作成時は例外フラグで許可）
  if (!context.allowEmpty && (content === '' || content == null)) {
    throw new ValidationError('V-1', 'empty content');
  }

  // V-2: 型検証
  if (typeof content !== 'string') {
    throw new ValidationError('V-2', `content is not string: ${typeof content}`);
  }

  // V-3: 異常な短縮検出（前回保存から 90% 以上縮んでいたら警告）
  if (context.previousLength && content.length < context.previousLength * 0.1) {
    throw new ValidationError(
      'V-3',
      `suspicious shrink: ${context.previousLength} -> ${content.length}`
    );
  }

  // V-4: NULL 文字・無効な UTF-16 サロゲートペアの検出
  if (/\u0000/.test(content)) {
    throw new ValidationError('V-4', 'NULL character detected');
  }
  if (hasLoneSurrogate(content)) {
    throw new ValidationError('V-4', 'lone surrogate detected');
  }
}
```

#### 2.2.2 V-3（異常短縮）は override 可能

意図的に大量削除した場合（例：章丸ごと削除）、V-3 に引っかかる。その場合は**ユーザー確認ダイアログ**を出す：

> 「作品が 89% 縮小しました。本当に保存しますか？
>   前回: 42万字 / 今回: 4万字
>   [保存する] [キャンセル（直前の状態に戻す）]」

「保存する」を選ぶと、**手動スナップショットを自動作成してから保存**する。これにより削除が意図通りでも、気が変わったら戻せる。

#### 2.2.3 V-1（空文字列）の override

新規ファイル作成時のみ、`allowEmpty: true` フラグで V-1 を通過できる。通常編集経路では絶対に空文字列が書き込まれない。

### 2.3 世代カウンタ：T-3 切替レース対策

ファイル A の autoSave が非同期で走っている最中にファイル B に切替。切替後に A の書き込みが完了すると、B のハンドル／パスに A の内容が書かれる可能性がある。これを防ぐ。

#### 2.3.1 仕組み

各ファイルハンドルに **世代カウンタ** `generation` を持たせる。ファイル切替のたびにカウンタが増える。

```js
// 擬似コード
let currentGeneration = 0;
let activeFileHandle = null;

function switchFile(newHandle) {
  currentGeneration += 1;
  activeFileHandle = newHandle;
  activeFileHandle.generation = currentGeneration;
}

async function autoSave(handle, content) {
  const startGen = handle.generation;
  // ... 非同期処理中に generation が変わっている可能性
  
  // 書き込み直前に再確認
  if (handle.generation !== startGen) {
    journal.warn('autoSave blocked: generation mismatch', {
      handle: handle.path,
      startGen, currentGen: handle.generation
    });
    return;
  }
  if (handle !== activeFileHandle) {
    journal.warn('autoSave blocked: handle no longer active');
    return;
  }

  await atomicWriteFile(handle.path, content);
}
```

#### 2.3.2 実装時の注意

既存 `useAutoSave.js` は `currentHandle !== activeFileHandle` で既にこの防御を入れている。ただし **generation 番号** で強化することで、同じハンドルオブジェクトが再利用されるエッジケース（モーダル経由で閉じて開き直した場合など）も防げる。

### 2.4 マニフェスト更新のトランザクション

マニフェスト更新（分割・統合・並び替え・リネーム）は複数ファイル操作を伴う。途中で止まると状態が不整合になる（T-4）。

#### 2.4.1 トランザクションログ方式

操作を「予定」と「確定」の 2 段階に分ける。

```
op-log/
├── pending/
│   └── tx-01936b9c-....json   ← これから行う操作の計画書
└── committed/
    └── tx-01936b9c-....json   ← 完了した操作の記録
```

手順：
1. pending/ に操作の計画書を atomic write で書き込む
2. 計画書に従って実ファイル操作を行う（すべて atomic write）
3. 各ステップ完了後に計画書のステータスを更新
4. 全ステップ完了したら計画書を committed/ に移動

NEXUS 起動時、pending/ に計画書が残っていたら **未完了トランザクション**と判定し、復旧フローに入る（§7）。

#### 2.4.2 計画書の例：セグメント分割

```json
{
  "txId": "01936b9c-a3c5-78e4-9c2a-...",
  "op": "segment.split",
  "workPath": "/Users/.../ノチユ.nexus",
  "status": "pending",
  "steps": [
    {
      "order": 1,
      "action": "snapshot.create",
      "target": "immediate/tx-01936b9c",
      "status": "pending"
    },
    {
      "order": 2,
      "action": "segment.write",
      "target": "segments/00001_第1章.txt",
      "contentHashExpected": "sha256:...",
      "status": "pending"
    },
    {
      "order": 3,
      "action": "segment.write",
      "target": "segments/00002_第2章.txt",
      "contentHashExpected": "sha256:...",
      "status": "pending"
    },
    {
      "order": 4,
      "action": "segment.rename_cascade",
      "from": ["00002_...", "00003_...", "..."],
      "to":   ["00003_...", "00004_...", "..."],
      "status": "pending"
    },
    {
      "order": 5,
      "action": "manifest.write",
      "target": "manifest.json",
      "contentHashExpected": "sha256:...",
      "status": "pending"
    }

```

#### 2.4.3 マニフェスト二重化

`manifest.json` は分割作品の生命線。これだけは必ず冗長化する：

- `manifest.json` 本体
- `manifest.json.backup`（前回の確定版）

書き込み手順：
1. atomic write で `manifest.json.new` を書く
2. 現 `manifest.json` を `manifest.json.backup` に rename
3. `manifest.json.new` を `manifest.json` に rename

起動時に `manifest.json` のパースに失敗したら `manifest.json.backup` を使う。両方壊れていたら、セグメントファイルから復元（§7.3）。

### 2.5 ジャーナル：すべての操作を記録

§3.4.2（設計書 A）で定義した `.nexus-project/journal.log` は、デバッグ用だけではなく、**復旧のための一次資料**として機能する。

#### 2.5.1 記録する操作

すべての書き込み・分割・統合・削除・リネーム・スナップショット作成・復元操作を記録する：

```jsonl
{"ts":"2026-04-19T15:23:11+09:00","level":"info","op":"atomic.write.begin","path":"segments/00001.txt","expectedHash":"sha256:a..."}
{"ts":"2026-04-19T15:23:11+09:00","level":"info","op":"atomic.write.success","path":"segments/00001.txt","hash":"sha256:a...","bytes":42180}
{"ts":"2026-04-19T15:23:12+09:00","level":"warn","op":"validation.V-3","path":"segments/00002.txt","previousLength":38000,"newLength":2000,"action":"prompted_user"}
{"ts":"2026-04-19T15:23:15+09:00","level":"error","op":"atomic.write.fail","path":"segments/00003.txt","error":"ENOSPC","phase":"writeFile"}
```

#### 2.5.2 ジャーナル自体の安全性

ジャーナル書き込みは O_APPEND でのみ行う。既存内容を書き換えることはない。これにより、ジャーナル書き込みでのバグが過去のログを消すことはない。

ジャーナル書き込みが失敗しても、本番操作は止めない（ログが欠ける程度の被害に留める）。

### 2.6 ブラウザ版（File System Access API）での対応

Electron でない環境（将来の Web 版、Tauri 移植前の中間状態）は、OS レベルの atomic rename が使えない。代替案：

- `FileSystemWritableFileStream` の transactional write（書き込み完了まで旧内容が見える）を使う
- スナップショットの頻度を Electron 版より高くして補う
- 大規模作品は Electron 版専用として案内する（ブラウザ版は 5 万字までを想定）

この設計書では Electron 版を主戦場と想定する。

## 3. スナップショット機構

### 3.1 4 層スナップショット

| 層 | 契機 | 保持 | 用途 |
|---|---|---|---|
| **immediate** | 編集のたび（debounce 付き、10 秒毎くらい） | 直近 20 版 | 「うっかり削除」からの即時復旧 |
| **hourly** | 1 時間毎 | 24 時間分 | 短期の巻き戻し |
| **daily** | 1 日毎（深夜、idle 時） | 7 日分 | 中期の巻き戻し |
| **pinned** | ユーザー手動 or マイルストーン | 無期限 | 賞応募版・完成版の保全 |

### 3.2 保存形式

スナップショットは作品全体を **1 ファイル** として保存する。セグメント形式のまま保存すると、復元手順が複雑になるため。

形式：**tar.zst 圧縮アーカイブ**（または zip）。

```
.nexus-project/snapshots/immediate/ノチユ/
├── 2026-04-19T15-23-11.tar.zst  ← manifest.json + segments/*.txt 全部入り
├── 2026-04-19T15-23-45.tar.zst
└── ...
```

内訳：
```
2026-04-19T15-23-11.tar.zst（展開後）
├── manifest.json
├── segments/
│   ├── 00000_プロローグ.txt
│   ├── 00001_第1章.txt
│   └── ...
└── snapshot-meta.json   ← このスナップショットのメタデータ
```

`snapshot-meta.json` 例：
```json
{
  "takenAt": "2026-04-19T15:23:11+09:00",
  "reason": "immediate-after-edit",
  "workId": "01936b9c-...",
  "totalChars": 42180,
  "segmentCount": 11,
  "overallChecksum": "sha256:..."
}
```

### 3.3 容量管理

#### 3.3.1 保持上限

| 層 | 件数上限 | 合計サイズ目安（42万字作品） |
|---|---|---|
| immediate | 20 | 約 20 MB（1 ファイル 1 MB 圧縮） |
| hourly | 24 | 約 24 MB |
| daily | 7 | 約 7 MB |
| pinned | 上限なし | ユーザー管理 |

1 作品あたり約 50 MB 程度。10 作品抱えても 500 MB。現代のディスク容量なら許容範囲。

#### 3.3.2 ガベージコレクション

NEXUS 起動時・毎日 1 回・手動操作時に、以下のルールで古いスナップショットを削除：

- immediate: 21 個目以降を古い順に削除
- hourly: 25 個目以降を古い順に削除
- daily: 8 個目以降を古い順に削除
- pinned: **削除しない**

削除は 30 日の grace period を置く：
- 削除対象となったスナップショットは `.nexus-project/snapshots/_trash/` に移動
- 30 日経ってから実際に削除
- grace period 中にユーザーが気付けば救える

#### 3.3.3 ディスク容量監視

スナップショット領域が 2 GB を超えたら、起動時にユーザー通知：

> 「スナップショット領域が 2GB を超えています。
>   古い immediate スナップショット 50 件を削除しますか？
>   [削除する] [手動で管理する] [後で]」

### 3.4 差分化（Phase 2 以降）

初期リリースは全文スナップショットで十分（圧縮後 1 MB/版）。容量が問題になったら、rdiff による差分スナップショットへ移行する。これは後続の最適化として扱い、初期リリースではシンプルさを優先する。

### 3.5 冗長配置

pinned スナップショットのみ、**作品ディレクトリ内**にも複製を置く：

```
ノチユ.nexus/
├── pinned-snapshots/    ← 作品と一緒に移動する重要版
│   └── 2026-10-31_submission.tar.zst
```

理由：プロジェクトフォルダごと移動・コピーした場合、`.nexus-project/snapshots/pinned/` は置き去りになる。賞応募版を pinned にしたあと、別ドライブへ作品ディレクトリをコピーしたときに応募版が失われることを防ぐ。

### 3.6 復元 UI

「時間を巻き戻す」画面で：

- 縦軸：時刻（新しい→古い）
- 各スナップショットを横長カードで表示
  - 日時
  - 総文字数（変化を矢印で可視化：`42180 → 42195 (+15)`）
  - 反映された変更の概要（journal から抽出）
  - プレビュー（冒頭 500 字）
  - [復元] [比較] [ピン留め] ボタン

復元操作は **現状を immediate スナップショットに保存してから** 実行される。「復元したら戻せなくなった」が起きない。

## 4. 複数インスタンス・ロック

### 4.1 ロックファイル

同じ作品を 2 つの NEXUS で開いたら、後から開いた方は **読み取り専用** になる。

```
ノチユ.nexus/
└── .lock
```

`.lock` ファイルの中身：

```json
{
  "pid": 12345,
  "host": "tonosuke-mbp",
  "user": "mokutagawa",
  "instanceId": "01936b9c-...",
  "acquiredAt": "2026-04-19T15:00:00+09:00",
  "lastHeartbeat": "2026-04-19T15:23:00+09:00"
}
```

### 4.2 ロック取得フロー

```
作品を開くとき:
1. .lock ファイルの存在確認
2. 存在する → lastHeartbeat を読む
   - 30 秒以内の heartbeat → 他インスタンスが生きている → 読み取り専用で開く
   - 30 秒以上経過 → 死んだインスタンスと判断 → 確認ダイアログを出してロック奪取
3. 存在しない → .lock を atomic write で作成
4. バックグラウンドで 10 秒毎に lastHeartbeat を更新
5. 作品を閉じるとき .lock を削除
```

### 4.3 ロック奪取ダイアログ

```
「この作品は別のウィンドウで開かれている可能性があります。
 最終応答: 5 分前
 [読み取り専用で開く（推奨）]  [強制的に編集モードで開く]
 
 注意: 強制モードで開くと、もう一方のウィンドウでの未保存編集は失われます」
```

### 4.4 読み取り専用モードの UI

- タイトルバーに「読み取り専用」バッジ
- textarea は `readOnly`
- 編集メニューはグレーアウト
- 他インスタンスが閉じたら自動的に編集モードへ昇格するかどうかを聞く

### 4.5 世代カウンタとの連携

ロックを取得できた場合でも、書き込みのたびに以下を確認：

1. .lock が自分のものか（ファイルを読み直して instanceId 確認）
2. 直前に読んだ manifest.json の checksum が変わっていないか
3. セグメントファイルの mtime が自分の知っている値と一致するか

1 つでも食い違えば書き込みを止めて、**競合解決ダイアログ**へ（§6.3）。

## 5. 分割・統合の境界検証

### 5.1 分割時の検証チェーン

分割操作は以下のチェーンを全部通る。1 つでも失敗したら分割を中止。

```
Step 1: 境界候補の検出
  意味境界 > 段落境界 > 文境界 > 強制 の優先順位で候補を生成

Step 2: 候補の安全性検証
  各候補について、前後のセグメントを独立にパース
  - 未閉合の《》、［＃］、{font}、**、[[ ]] を検出
  - 1 つでもあれば、その候補は却下

Step 3: ユーザー承認 UI（F-11.5）
  前後 400 字を並べて表示
  ユーザーが承認 → Step 4 へ
  ユーザーが「少しずらす」→ 代替候補を再生成
  ユーザーが却下 → 分割中止、セグメントはそのまま

Step 4: ラウンドトリップ検証
  分割前: 単一セグメント S
  分割後: S1 + S2
  検証: S === S1 + separator + S2 か?
        （separator は空文字または 1 つの \n。事前に決めたルールで一意）
  失敗したら分割中止

Step 5: スナップショット作成
  分割前の状態を immediate スナップショットに保存

Step 6: トランザクションログ書き込み（§2.4）
Step 7: atomic write で各ファイル書き込み
Step 8: マニフェスト更新
Step 9: 完了をジャーナルに記録
```

### 5.2 統合時のラウンドトリップ検証

2 つのセグメント A, B を統合して C にする：

```
C = A + separator + B

検証:
  C を分割アルゴリズムに通し、A と B が再現できるか?
  → 再現できれば OK
  → 再現できない場合（意味境界が消えた等）はユーザーに警告し、
    「このまま統合すると再分割時に境界位置が変わるかもしれません」と表示
```

統合前の状態もスナップショットに保存してから操作を実行する。

### 5.3 境界のユーザー承認記録

ユーザーが「ここで分けていい」と承認した境界は manifest.json に記録：

```json
{
  "segments": [
    {
      "id": "s-00001",
      "boundaryToPrev": {
        "approvedAt": "2026-04-19T15:23:11+09:00",
        "approvedBy": "user",
        "marker": "■",
        "contextBefore": "...そして戦いは終わった。\n\n",
        "contextAfter": "■第二章 嵐の前触れ\n\n..."
      }
    }
  ]
}
```

同じ作品で再度分割提案する際、過去に承認された境界パターン（marker の種類、前後コンテキストの類似度）を優先候補として提示する。

## 6. 外部改変の検出

### 6.1 起動時検証

作品を開いたとき：

1. `manifest.json` を読む
2. 各セグメントファイルについて、`mtime` と `checksum` をマニフェストと比較
3. いずれか食い違えば「外部改変検出」として扱う

### 6.2 外部改変検出時の UI

```
「外部から変更された可能性があります。

 ファイル: segments/00003_第3章.txt
 マニフェスト上の更新時刻: 2026-04-19 10:00
 ファイルシステム上の更新時刻: 2026-04-19 14:30
 文字数: 42,180 → 41,500 (-680)

 選択肢:
   [外部変更を受け入れる]
     ファイルシステム上の内容を正とし、マニフェストを更新します
     現在のマニフェストはスナップショットに保存されます
 
   [マニフェストに合わせて復元する]
     最新スナップショットから復元します
     ファイルシステム上の変更は失われます（スナップショットに保存）
 
   [後で決める]
     読み取り専用で開きます」
```

どれを選んでも、実行前に**現状のスナップショット**が自動作成される。決断は後からでも取り消せる。

### 6.3 書き込み直前の mtime 再確認

書き込み直前に、**そのセグメントの mtime が、自分が最後に読んだときと同じか**を確認する。違っていたら、他プロセスか外部ツールが書き換えた証拠。

違っていた場合：
1. 書き込みを停止
2. 新しい内容をスナップショットに保存
3. 競合解決ダイアログを表示

## 7. 復旧フロー

### 7.1 NEXUS 起動時の自己診断

起動のたびに以下を実行：

```
1. .nexus-project/op-log/pending/ を走査
   → 未完了トランザクションがあれば §7.2 へ

2. 各作品の manifest.json を読む
   → パース失敗なら manifest.json.backup を試す
   → 両方ダメなら §7.3 へ

3. 各セグメントの mtime + checksum を検証
   → 食い違いがあれば §6.2 の外部改変ダイアログ

4. ロックファイルの検証
   → 自分のインスタンスの残骸（前回クラッシュ）なら自動削除
   → 他インスタンスなら §4.2

5. スナップショット領域の整合性チェック
   → 破損スナップショットを検出したら _trash/ へ退避
```

### 7.2 未完了トランザクションの復旧

pending/ に計画書が残っている場合：

```
1. 計画書を読む
2. status が "pending" の最初のステップを特定
3. そこまでは完了している可能性と、完了していない可能性がある
4. 現状のファイルシステムを確認し、実際の状態を判定
5. 以下のいずれかを選択（ユーザーに提示）:
   a) トランザクションをロールバック
      → 計画書にロールバック手順が書かれていれば実行
      → なければ、トランザクション開始直前のスナップショットから復元
   b) トランザクションを再開
      → 残りのステップを実行
   c) 手動で解決
      → 読み取り専用で開き、ユーザーがファイルを見て判断
```

推奨はロールバック。再開は計画書の整合性がとれている場合のみ提示。

### 7.3 マニフェスト全損時のセグメント再構築

`manifest.json` と `manifest.json.backup` の両方が失われた場合：

```
1. segments/ 配下のファイル一覧を取得
2. ファイル名の通し番号順に並べる
3. 各ファイルの charCount と checksum を計算
4. スナップショットから過去の manifest.json を探す
5. 最新のスナップショットマニフェストをベースに、現在のセグメントで更新
6. 再構築したマニフェストを提示して、ユーザーに承認を求める
7. 承認されたら atomic write で書き戻す
```

最悪のケースでも、**セグメントファイル自体は無事**なら作品は復旧できる。これが設計書 A で「セグメントをただの .txt」にした最大の理由。

### 7.4 全滅時の最終手段

作品ディレクトリ全体が失われた場合：

1. プロジェクトの `.nexus-project/snapshots/` を確認
2. 該当作品のスナップショットがあれば、そこから作品ディレクトリを再構築
3. ユーザー確認のうえ復元

プロジェクト自体が失われていたら、OS レベルのバックアップ（Time Machine、iCloud）に頼るしかない。NEXUS の責任範囲外。

## 8. 5 日分飛んだ事故の再現防止チェックリスト

冬之助さんの 2026 年 4 月事故を、この設計で再現できるか検証する。事故の推定経路：

> パフォーマンス改善中にコード変更で保存ロジックが壊れ、空文字列かゴミ文字列が本番ファイルに書き込まれた。Time Machine や iCloud の復元はできなかった（取っていなかったか、同期されて壊れた状態のまま保存された）。

### 事故再現シナリオ A：空文字列書き込み

| ステップ | この設計での挙動 |
|---|---|
| 1. localText が undefined の瞬間に onChange が発火 | V-1 が検出して throw |
| 2. autoSave が受け取る | 例外発生、保存されない |
| 3. 上位で catch | トーストで「保存失敗」を表示 |
| 結果 | **本番ファイル無事** |

### 事故再現シナリオ B：異常短縮

| ステップ | この設計での挙動 |
|---|---|
| 1. 編集ロジックバグで 42 万字 → 4 千字に一気に縮む | V-3 が検出 |
| 2. 保存が進もうとする | ユーザー確認ダイアログが出る |
| 3. ユーザーが気付く | キャンセルを押せば直前状態に戻る |
| 4. ユーザーが気付かず承認しても | 手動スナップショットが自動作成されてから保存 |
| 結果 | **スナップショットから即復旧可能** |

### 事故再現シナリオ C：書き込み中クラッシュ

| ステップ | この設計での挙動 |
|---|---|
| 1. atomic write 中にクラッシュ | 本番ファイルは元のまま |
| 2. .tmp.* ファイルが残骸として残る | 次回起動時のクリーンアップで削除 |
| 結果 | **本番ファイル無事** |

### 事故再現シナリオ D：マニフェスト中断

| ステップ | この設計での挙動 |
|---|---|
| 1. 分割中にクラッシュ | pending/ に未完了トランザクション |
| 2. 次回起動時 | 自己診断で検出、§7.2 で復旧提示 |
| 3. ユーザー承認でロールバック | スナップショットから復元 |
| 結果 | **分割前の状態で無事** |

### 事故再現シナリオ E：過去の事故そのもの

事故当時の具体的な原因は特定できていないが、**どんな経路であれ**、以下のすべてを通過するまで本番ファイルには書き込まれない：

1. V-1（空チェック）
2. V-3（短縮チェック）
3. atomic write の事前パス検証
4. 一時ファイルへの書き込み
5. 書き込み後のチェックサム検証
6. atomic rename
7. ジャーナル記録

かつ、どの時点でも：
- immediate スナップショット（直近 10 秒単位）
- hourly スナップショット（直近 24 時間）
- daily スナップショット（直近 7 日）

が存在する。5 日分がまとめて消えるには、これらすべてが同時に壊れる必要があり、構造的にほぼ不可能。

## 9. この設計で守られること・守られないこと

### 守られること

- 書き込み途中のクラッシュで半端なファイルが残らない
- 空文字列・極端な短縮は検出される
- 書き込みペイロードの同一性が読み直し検証される
- 複数インスタンスが原稿を殴り合わない
- 境界計算ミスでセグメントが壊れても、分割前の状態に戻せる
- 外部改変は検出され、スナップショットを取ってから対処できる
- マニフェストが壊れてもセグメントから復元できる
- 最低 7 日（immediate + hourly + daily）、ピン留めすれば無期限の履歴がある
- 全破壊操作はジャーナルで追跡可能
- 「5 日分飛んだ」事故は構造的に再現不可能

### 守られないこと

- OS レベルのディスク故障・SSD 寿命・ビット腐敗
  - Time Machine、iCloud、Backblaze 等の**外部バックアップは引き続き必要**
- ユーザーが作品ディレクトリごと手動削除した場合
  - プロジェクトのスナップショットに残るが、プロジェクトごと消えたら復旧不能
- NEXUS のバイナリ自体が破損した場合
  - 再インストールで復旧可能だが、その間編集不可
- ブラウザ版での完全 atomic write（OS サポートがないため）
  - Electron 版を主戦場とする
- `_trash/` の 30 日 grace period を過ぎた自動削除
  - 手動で `_trash/` から戻すオプションは提供するが、自動では戻らない

### 後続で扱うこと

- **UI での復元操作・スナップショット閲覧**：設計書 C
- **実装フェーズと既存 NEXUS コードの書き換え順序**：設計書 D
- **差分スナップショット最適化**：Phase 2 以降
- **Tauri 移植時の atomic write 互換**：Tauri 移植計画（別ドキュメント）
