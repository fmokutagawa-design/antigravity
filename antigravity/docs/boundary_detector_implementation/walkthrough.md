# 指示書 C（章境界検出ユーティリティの実装）実装完了レポート

## 概要

NEXUS エディタが 42万字などの超巨大なテキストを扱う際、裏側で「章」や「節」などの意味的境界をパースし、文章を安全にチャンク分割するための純粋関数群（Boundary Detector）を実装しました（ブランチ `feature/boundary-detector`）。

この機能は、現状ではまだどこの UI ロジックとも結合していない「純粋なユーティリティ」ですが、将来的な透明分割機構（Transparent Segmentation）の最重要の基礎となります。

## 実装内容

### 1. 境界候補の検出 (`findBoundaryCandidates`)
*   **ファイル**: `src/utils/boundaryDetector.js`
*   テキストの中から、以下の見出しマーカーを全件列挙し、信頼度（Confidence）付きで抽出する正規表現パターンを実装しました。
    *   **「■」から始まる行**: 章 (`chapter`) - 信頼度 0.95
    *   **「第〇〇章」「第〇〇話」の行**: 章 (`chapter`) - 信頼度 0.9
    *   **Markdown見出し (`#`, `##`...)**: 節 (`section`) - 信頼度 0.85
    *   **青空文庫の大・中・小見出し (`［＃大見出し］...`)**: 章または節 - 信頼度 0.75〜0.95
    *   **青空文庫の改ページ (`［＃改ページ］`)**: 強制分割点 (`aozora-pagebreak`) - 信頼度 0.9
    *   **空行の3連続以上**: 段落の切れ目 (`paragraph`) - 信頼度 0.4
*   同一箇所で複数のルールが衝突し得る場合（例: ［＃改ページ］と重なる等）、最も信頼度の高いものを一つだけ残す deduplicate 処理を組み込みました。

### 2. 境界の安全性検証 (`validateBoundary`)
*   **ファイル**: 同上
*   指定された offset（文字インデックス）でテキストを真っ二つに切った際、特殊なメタ文字や構文を破壊しないか検証するパーサを実装しました。
    *   **サロゲートペア**: 文字化け防止のため、サロゲートペアの中間で切ることを禁止
    *   **ルビ記号**: `《...》` の内部での分割を禁止
    *   **青空文庫タグ**: `［＃...］` の内部での分割を禁止
    *   **内部フォントタグ**: `{font: ...}` 〜 `{/font}` ブロック内での分割を禁止
    *   **その他 Markdown 修飾**: 強調 (`**...**`)、Wikiリンク (`[[...]]`、`［［...］］`) 内部での分割を禁止

### 3. テストの網羅
*   **ファイル**: `src/utils/boundaryDetector.test.cjs`
*   Node.js ネイティブ環境で `ESM` 実装を動的 Import してテストするスクリプトを用意し、**全23件** のすべてのエッジケース（重複検知、タグの中間等）をクリアすることを確認しました。
